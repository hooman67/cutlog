# Migration Plan: CutLog Web App to Native (iOS + Android) via Capacitor

## 1. Executive Summary

**Goal:** Publish CutLog on the Apple App Store and Google Play Store by wrapping the existing Next.js web app with Capacitor — no rewrite required.

**Why Capacitor (not React Native / Flutter)?**

| Factor | Capacitor | React Native / Flutter |
|--------|-----------|----------------------|
| Code reuse | 95%+ of existing code unchanged | Full rewrite required |
| Development time | 1-2 weeks | 2-4 months |
| Cost | ~$300 (dev accounts only) | $10,000-40,000+ (dev time) |
| Risk | Low — web app continues working | High — new bugs, new stack |
| Maintenance | One codebase | Two codebases to maintain |

**Why now (Month 2-3)?** Once we have 5+ active users with validated daily usage patterns, a native app provides:
- Push notifications for parameter recommendations
- App Store discoverability (ASO)
- Perceived legitimacy ("real app" vs bookmark)
- Camera access for .clb file scanning
- Offline mode for shop floor use without WiFi

**Estimated timeline:** 5-8 working days total
**Estimated cost:** ~$324 (Apple $99/yr + Google $25 one-time + misc)

```
|  |  +--------------------------------------+  |  |
|  |  |        CutLog Web App               |  |  |
|  |  |   (Next.js static export OR         |  |  |
|  |  |    hosted at cutlog-two.vercel.app) |  |  |
|  |  +--------------------------------------+  |  |
|  |                                            |  |
|  +--------------------------------------------+  |
|                                                    |
|  +--------------------------------------------+  |
|  |        Capacitor Plugin Bridge             |  |
|  |   (Camera, Push, Haptics, Storage, etc.)   |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
```

### What Stays Web vs What Becomes Native

| Layer | Stays Web | Becomes Native |
|-------|-----------|---------------|
| UI / Components | All React components, Tailwind CSS | - |
| Routing | Next.js pages (/, /log, /suggest, etc.) | - |
| Auth | Supabase Auth (email/password, OAuth) | Deep links for OAuth callback |
| Database | Supabase client calls (unchanged) | - |
| API | Supabase direct (no Next.js API routes needed in static mode) | - |
| Push Notifications | - | Capacitor Push Plugin |
| Camera | - | Capacitor Camera Plugin |
| Haptic Feedback | - | Capacitor Haptics Plugin |
| Offline Storage | - | Capacitor Preferences / SQLite |
| App Lifecycle | - | Capacitor App Plugin |

## 4. Step-by-Step Implementation Plan

### Phase 1: Project Setup (Day 1)

#### 1.1 Install Capacitor Core

```bash
cd /mnt/localssd/laser_log/app

# Install Capacitor
npm install @capacitor/core
npm install -D @capacitor/cli

# Initialize Capacitor config
npx cap init "CutLog" "com.cutlog.app" --web-dir=out
```

This creates `capacitor.config.ts` in the project root.

#### 1.2 Configure Capacitor

Create/edit `capacitor.config.ts`:

```typescript
import type { CapacitorConfig } from '@capacitor/cli';

const config: CapacitorConfig = {
  appId: 'com.cutlog.app',
  appName: 'CutLog',
  webDir: 'out',  // Next.js static export output directory
  server: {
    // For development, point to local dev server:
    // url: 'http://192.168.x.x:3000',
    // cleartext: true,
    
    // For production: comment out url to use local static files
    // OR use hosted URL:
    // url: 'https://cutlog-two.vercel.app',
  },
  plugins: {
    SplashScreen: {
      launchShowDuration: 2000,
      backgroundColor: '#1a1a2e',  // Match app dark theme
      showSpinner: false,
    },
    StatusBar: {
      style: 'dark',
      backgroundColor: '#1a1a2e',
    },
  },
};

export default config;
```

#### 1.3 Configure Next.js for Static Export

Edit `next.config.mjs`:

```javascript
/** @type {import('next').NextConfig} */
const nextConfig = {
  eslint: {
    ignoreDuringBuilds: true,
  },
  output: 'export',  // Generate static HTML/CSS/JS in /out
  // If using images, add:
  images: {
    unoptimized: true,  // Required for static export
  },
};

export default nextConfig;
```

#### 1.4 Build Static Export

```bash
npm run build
# This outputs to /out directory when output: 'export' is set
```

#### 1.5 Add Native Platforms

```bash
# Add iOS
npm install @capacitor/ios
npx cap add ios

# Add Android
npm install @capacitor/android
npx cap add android
```

This generates:
- `ios/` — Xcode project
- `android/` — Android Studio project

#### 1.6 Sync Web Assets to Native Projects

```bash
npx cap sync
```

Run this after every web build to copy `/out` into native projects.

#### 1.7 Add Build Script

Add to `package.json` scripts:

```json
{
  "scripts": {
    "build:mobile": "next build && npx cap sync",
    "open:ios": "npx cap open ios",
    "open:android": "npx cap open android"
  }
}
```

```typescript
// src/lib/push-notifications.ts
import { PushNotifications } from '@capacitor/push-notifications';
import { Capacitor } from '@capacitor/core';

export async function initPushNotifications() {
  if (!Capacitor.isNativePlatform()) return;

  const permission = await PushNotifications.requestPermissions();
  if (permission.receive !== 'granted') return;

  await PushNotifications.register();

  PushNotifications.addListener('registration', (token) => {
    // Send token to Supabase to store for this user
    console.log('Push token:', token.value);
    // saveTokenToSupabase(token.value);
  });

  PushNotifications.addListener('pushNotificationReceived', (notification) => {
    console.log('Push received:', notification);
  });

  PushNotifications.addListener('pushNotificationActionPerformed', (action) => {
    // Handle tap on notification — navigate to relevant page
    console.log('Push action:', action);
  });
}
```

```typescript
// src/lib/offline-cache.ts
import { Preferences } from '@capacitor/preferences';

export async function cacheParameters(params: any[]) {
  await Preferences.set({
    key: 'cached_parameters',
    value: JSON.stringify(params),
  });
}

export async function getCachedParameters() {
  const { value } = await Preferences.get({ key: 'cached_parameters' });
  return value ? JSON.parse(value) : null;
}
```

## 5. Next.js + Capacitor Specifics

### The SSR Problem

Next.js 14 uses Server-Side Rendering (SSR) and Server Components by default. Capacitor needs either:
- A) Static files it can bundle into the app (no server needed)
- B) A URL pointing to a hosted version

### Making Next.js Work with Static Export

Key changes needed:

1. **Add `output: 'export'` to `next.config.mjs`**

2. **Remove/modify API routes** (`src/app/api/`)
   - Current API routes won't work in static mode
   - Move all data fetching to client-side Supabase calls (already mostly done)
   - If any API routes exist only for server-side logic, refactor to Supabase Edge Functions

3. **Convert Server Components to Client Components where needed**
   - Any component using `cookies()`, `headers()`, or dynamic server features needs `'use client'`
   - Components that only fetch from Supabase client-side are already fine

4. **Handle dynamic routes**
   - Use `generateStaticParams()` for dynamic routes, or
   - Convert to client-side routing with `useSearchParams()`

5. **Remove `next/image` optimization** (or set `unoptimized: true`)
   - Next.js Image Optimization requires a server
   - Use standard `<img>` tags or set `unoptimized: true` in config

### Checking Current API Route Usage

The current app has an `/api` directory. Review each route:
- If it's just proxying Supabase calls: remove and call Supabase directly from client
- If it has server-only logic: move to Supabase Edge Functions

## 6. Two Approaches: Static Export vs Hosted URL

### Approach A: Static Export (Recommended)

```
next.config.mjs: output: 'export'
capacitor.config.ts: webDir: 'out' (no server.url)
```

**How it works:**
- `npm run build` generates static HTML/CSS/JS in `/out`
- `npx cap sync` copies `/out` into native app bundles
- App runs entirely from local files + Supabase API calls

**Pros:**
- Works offline (with cached data)
- Instant load (no network request for UI)
- No dependency on Vercel being up
- Better performance (no WebView loading remote page)
- Apple prefers this (less likely to be rejected as "just a website")

**Cons:**
- Need to rebuild and submit app update for UI changes
- API routes don't work (must use client-side Supabase)
- Slightly more complex build pipeline

### Approach B: Hosted URL (WebView to Vercel)

```
capacitor.config.ts: server: { url: 'https://cutlog-two.vercel.app' }
```

**How it works:**
- Native app opens a WebView pointed at the live Vercel URL
- All rendering happens server-side as normal
- App is essentially a branded browser

**Pros:**
- Zero changes to existing code
- Updates deploy instantly (no app store review for web changes)
- API routes continue working
- Server Components work as-is

**Cons:**
- Requires internet to function at all
- Slower load (full page fetch on open)
- Higher risk of Apple rejection ("why not just use Safari?")
- No offline capability
- User experience identical to "Add to Home Screen"

### Recommendation: Approach A (Static Export) with Hybrid Strategy

**Use static export as the primary approach**, but with a fallback architecture:

1. Build the app statically for all UI and client-side logic
2. Keep Supabase calls as client-side (they already work this way)
3. For any complex server logic, use Supabase Edge Functions (not Next.js API routes)
4. Cache parameters locally for offline use
5. If a web-only update is urgent, push a config flag via Supabase that shows an "Update Available" prompt

This gives you the best of both worlds:
- Native speed and offline capability
- Pass App Store review (it's a real app, not a web wrapper)
- Can still update data/recommendations server-side without app updates

## 10. Trigger Criteria

### When to Start This Work

**Hard requirements (ALL must be true):**

1. **5+ active users** with accounts who have logged cuts in the past 7 days
2. **Daily usage validated** — at least 3 users returning to the app on consecutive days
3. **Core features stable** — no critical bugs in log, suggest, or history features
4. **Supabase auth working via client-side** — no dependency on Next.js API routes for auth

**Soft signals (nice to have):**

- Users asking "Is there an app for this?"
- Add to Home Screen usage detected (if measurable)
- Feature requests that require native capabilities (camera, offline)
- Competitor launches native app

### Decision Checklist

```
[ ] 5+ users with activity in past 7 days
[ ] 3+ users with multi-day return visits
[ ] No P0 bugs open
[ ] Auth works fully client-side
[ ] All data operations use Supabase client (not API routes)
[ ] Apple Developer account enrolled

```

### What to Do Before the Trigger

Even before starting native development, these prep tasks reduce Day 1 friction:

1. **Enroll in Apple Developer Program** (takes 1-2 days for approval)
2. **Enroll in Google Play Developer** (takes 1-2 days for approval)
3. **Ensure all data operations are client-side Supabase calls** (no server-side only logic)
4. **Test `next build` with `output: 'export'`** — fix any issues early
5. **Design app icon** (1024x1024 PNG)
6. **Remove any dependency on Next.js API routes** or migrate them to Supabase Edge Functions
